ToolRouter endpoint onboarding map

Adding a new research service

This page shows what changes when ToolRouter adds a new provider-backed research endpoint. The example is hypothetical: researchco.research, a service that accepts a topic and returns cited research results.

Runtime Code Path

01

Agent chooses endpoint

MCP lists categories and endpoints. The agent explicitly calls a concrete id such as researchco.research.

02

API receives call

POST /v1/requests authenticates the ToolRouter API key and looks up the endpoint in the registry.

03

Builder validates input

The endpoint builder converts typed user input into a provider request and an estimated USD cost.

04

Executor pays

The executor tries AgentKit first, then x402 fallback when needed. It records path, charge state, status, latency, and errors.

05

Credits finalize

The API captures actual spend or releases unused reserved credits, then writes the request trace.

06

Dashboard and traces

Customer dashboard and MCP trace lookup read the stored request row. No provider secrets reach the browser.

Files You Touch

builders.ts

Add buildResearchCoResearchRequest(input, endpoint). It validates the public input shape and returns { method, url, headers, json, estimatedUsd }.

endpoints/<category>/...

Add a provider endpoint definition with metadata, AgentKit value type, fixture input, health probe, live smoke config, and the builder.

registry.ts

Import the new definition and add it to ENDPOINT_DEFINITIONS. Registry validation then makes it visible to API, MCP, health worker, tests, and dashboard metadata.

categories.ts

Use an existing category such as search or data, or add a new category only if agents need a distinct discovery bucket.

apps/mcp/src/server.ts

Generic toolrouter_call_endpoint works automatically. Add a convenience MCP tool only if the endpoint deserves a first-class alias.

tests/**

Add deterministic registry/builder tests and opt-in live smoke tests. Paid tests stay behind explicit env gates and strict maxUsd.

builder 
export function buildResearchCoResearchRequest(input, endpoint) {
  const data = assertInputRecord(input);
  const topic = readString(data, ["topic", "query"], "topic", { required: true });
  const depth = readString(data, ["depth"], "depth", { defaultValue: "fast" });
  const maxSources = readInteger(data, ["maxSources", "max_sources"], "maxSources", {
    defaultValue: 5,
    min: 1,
    max: 10
  });

  return providerRequest(
    endpoint,
    { topic, depth, maxSources },
    depth === "deep" ? 0.05 : 0.02
  );
}

AgentKit Benefit Decision

free_trial

AgentKit-Free Trial

Use when a verified human delegation can call the provider without payment until a trial quota is exhausted.

discount

AgentKit-Discount

Use when AgentKit proof lowers the price, but the call still pays through x402.

access

AgentKit-Access

Use when AgentKit proof unlocks access, even though payment still happens through x402.

What Becomes Automatic

Automatic after registry import

  • GET /v1/endpoints includes the endpoint metadata.
  • GET /v1/categories includes it under its category.
  • POST /v1/requests can execute it by endpoint id.
  • The health worker can build its probe request from health_probe.
  • The generic MCP tool can call it through toolrouter_call_endpoint.

Manual choices

  • Whether the endpoint gets a dedicated MCP convenience tool.
  • Whether it becomes the recommended endpoint for a category.
  • Which official provider logo to save for UI surfaces.
  • Which live smoke flags and cost caps to add.
  • Which AgentKit benefit type it realizes.

Test Checklist

  1. Registry validation passes. The new endpoint has all required metadata, fixture, health probe, live smoke config, and builder.
  2. Builder unit test passes. Inputs map to the exact provider JSON shape and estimated cost.
  3. Category test is updated. The endpoint appears under the intended discovery category and recommendation behavior is explicit.
  4. MCP test is updated only if a dedicated tool is added. The generic call endpoint needs no new tool test.
  5. Live smoke is opt-in. Use env gates, fixture input, strict maxUsd, and assert the expected path and charge behavior.
